Architecture

Dub is a Rust audio engine wrapped by a native macOS SwiftUI / AppKit shell. The Rust core owns the audio thread end-to-end; Swift owns the UI thread end-to-end. They communicate via lock-free state snapshots and SPSC ring buffers, never callbacks across thread boundaries.

15 Rust crates · 1 SwiftUI shell · 3-layer RT-safety enforcement

Process overview

One macOS process. Two long-lived threads matter: the SwiftUI main actor (UI), and the CoreAudio AU IO proc (RT). All cross-thread communication is lock-free.

Crate dependency graph

Two things to take from this graph that the boxes alone don't say: only dub-engine runs on the audio thread, and dub-cli depends directly on the engine, not through the FFI (it's the headless test harness, never crosses UniFFI).

RT-safety enforcement

Three layers, in order of strength. Each catches a strictly larger class of violations than the one above; the audio thread is statically guarded at the type level, and runtime allocators trap any escape that the type system did not catch.

UI ↔ Engine messaging

Bidirectional and lock-free. The UI sends typed commands through a SPSC ring buffer; the engine sends displaced heap-bearing payloads back through dedicated trash channels so Arc::drop and Box::drop run on the main thread instead of the audio thread.

UI → Engine commands (hot path) Engine → UI state snapshot (warm) Engine → UI trash channels (cold heap-back)

HAL input invariant — sample-rate match

CoreAudio HAL has a load-bearing footgun: if the AudioUnit's stream format SR does not equal the device's hardware nominal SR, the IO proc silently delivers zero callbacks. AudioUnitStart returns OK, coreaudiod logs nothing, the mic indicator never lights up. AudioInput::start_with_options enforces the invariant in this exact sequence.
