Beat grid · tempo · waveform contract

Beat grid, tempo and the waveform

Everything Dub does about tempo and beat phase. The data model is three fields; the waveform draws what the data model says; the user has six gestures. This page exists so the team and the docs all use the same vocabulary.

v0.1 working spec · 2026-05-24 · Replaces PRD §8.3.1 · PRD-BEATS.md

1 · Anatomy of a beat grid

A beat grid is three fields: bpm, beat_anchor_secs, and bar_phase. Everything else is derived. The grid is continuous, uniform, and covers the entire track — including silence. The yellow downbeats are a separate property of the grid; rotating them does not move beats.

Read the picture this way: short grey ticks above the waveform are beats; the full-height yellow lines through the waveform are downbeats (Serato-style rendering — see §5 for the contract). The grid runs from t = 0 to t = 8 s; the audio only plays between 1.5 s and 6.5 s. The first downbeat at t = 0 is in silence — that is correct and expected. The next three downbeats (at 2, 4, 6 s) land on the audible kicks, which is the visible payoff. The waveform is clipped vertically so the ticks and downbeat marker are always visible, even on busy tracks.

2 · Vocabulary

Binding for Rust code, FFI structs, Swift UI copy, log lines, error messages, and bug reports. Defined once here, referenced everywhere else.

BPMTempo in beats per minute. Scalar, positive.
periodSeconds per beat = 60 / BPM. Derived; never stored.
beatOne tick on the grid. A single point in time.
bar / measureA group of N consecutive beats. v1 hard-codes N = 4.
bar positionInteger 1..4 identifying a beat's place inside its bar.
downbeat / "the 1"Beat whose bar position is 1.
beat_anchor_secsOne known beat's timestamp. All other beats derived: beat[i] = anchor + i × period.
bar_phaseInteger 0..3 saying which beat in every group of 4 is the downbeat. Independent of beat_anchor_secs.
gridThe full uniform set of beat timestamps for one track, clipped to [0, duration].
driftMeasured deviation between grid beats and audible transients, accumulating over time.
grid lockUser-blessed frozen grid. Absolute: no analyze, re-analyze, tap-tempo, or set-the-1 mutates a locked grid. The user must explicitly unlock first.
library rowThe active track_beatgrids row for a track. Single source of truth for BPM in both the library list cell and the deck header.
waveform sidecarPre-rendered peaks file at ~/Library/Caches/Dub/waveforms/{fingerprint}.wf. Written by analyze passes, read on deck load so the waveform paints before the first frame.
ODFOnset-detection function. Per-band spectral-flux signal that peaks at musical transients.
transient snapRefining a user-supplied time onto the nearest real transient inside a bounded window.
noise floor (snap)Minimum ODF magnitude a peak must clear to count as real. Adaptive: 12% of track-wide p95 ODF.

3 · The data model

Three fields. Two are stored, one is integer phase, and the rest of the world (renderer, FX, library) reads only derived data. 

struct BeatGrid {
    bpm:              f64,    // tempo, > 0
    beat_anchor_secs: f64,    // one known beat's time
    bar_phase:        u8,     // 0..3, which beat is the downbeat
}

beats     = { beat_anchor_secs + i × (60 / bpm)
              for all integer i, clipped to [0, duration] }
downbeats = { beats[i] where i mod 4 == bar_phase }

What this buys us

  • Continuous grid across the entire track. No "where does the grid start" debate.
  • bar_phase rotates without moving beats.
  • Renderer never infers downbeats from beats[0].
  • Lock / unlock flips a flag; the data is unchanged.
  • Library round-trip is three numbers per track, plus the lock flag.

What it forbids

  • Trimming beats around silence. Beats cover [0, duration], period.
  • Inferring downbeats from "first audible beat." Bar phase is explicit.
  • Tempo curves. One BPM per track.
  • Half-bars, swing, or time-signature changes. v1 is strict 4/4.

5 · Set the 1 — pure bar-phase rotation

The most-misunderstood gesture. One tap rotates which 1-in-4 beats are yellow. Beat positions do not change, BPM does not change, the grid does not shift. The user's tap defines which existing beat is "the 1"; everything else is identical to before.

What changed: only bar_phase. Every beat is in the exact same place. The yellow rotated right by one beat because the nearest beat to the user's tap (at 2.5 s, beat index 5) is now the downbeat, and 5 mod 4 = 1. BPM, beat anchor, and waveform are untouched.

6 · Tap tempo — constrained re-analysis driven by the user's hint

Three or more taps inside the 2 s window tell the algorithm where to look: a ±15 % BPM window around the tap median. The full estimator runs again inside that window; the strongest real autocorrelation peak is the answer. First tap snaps to the nearest transient to seed the anchor; bar_phase is the value that best fits the user's tap times across the four candidates. The tap is never the BPM — a 2 s window of 3–8 taps cannot beat a full-track spectral-flux estimator on precision (human reaction-time jitter is ~25 ms 1σ per tap edge). The taps disambiguate; the algorithm answers.

Tap 1 lands a few ms past the kick because of human reaction time plus audio latency. The transient snap (12 % p95 noise floor) pulls it onto the actual kick. The four taps fix the BPM neighborhood; the algorithm then runs constrained autocorrelation over the full track inside that window.

Constrained re-analysis (M11d.7 round 3)

The tap median is a search hint, not the answer. The full estimator runs again with the search range pinned to the tap-median neighborhood; the strongest real ODF peak in that window is the BPM, snapped to integer if residuals don't get worse. 

tap_median   = weighted_median(60 / interval(taps))     // outlier-filtered

# ±15 % around tap median. Wider than human tap noise
# (~5 BPM 1σ at 100 BPM); tight enough to exclude
# half/double octaves (at ±50 % / +100 %).
search_range = BpmRange::new(tap_median * 0.85, tap_median * 1.15)

(bpm_raw, broadband_odf, kick_odf) =
    analyze_bpm_with_range_profile_and_odfs(
        samples, sample_rate, channels, search_range, profile)

bpm_final = snap_to_integer_if_safe(bpm_raw, residuals)
new_anchor = snap_to_nearest_transient(first_tap, kick_odf, broadband_odf,
                                        noise_floor = 12% of p95)
bar_phase  = argmin_p Σ |t_i − nearest_downbeat(t_i, anchor, period, p)|²
                                                       // p ∈ {0,1,2,3}
ScenarioWhat the algorithm seesResult
Auto says 109.3, user taps 108 (small error) Search 91.8–124.2; the strongest peak is still 109.3 109.3 unchanged (the user's slight misread doesn't disturb a good auto BPM)
Auto says 87 (octave error), user taps 174 Search 148–200; auto's wrong-octave peak is excluded by construction 174.3 → 174 after integer snap (the user's tap unstuck the algorithm from the wrong octave)

The previous reconcile_tap_bpm_with_hint with its ±5 % "keep hint" band and ±5 % "octave detect" branches is removed entirely. One rule replaces four cases; octave errors are handled structurally by the search-range constraint rather than by a special branch.

Session window — dynamic, not 2 seconds

A tap session opens on the first tap. Subsequent taps belong to the same session as long as they arrive within max(1.5 s, 1.5 × median tap interval so far) of the previous tap. This is a dynamic ceiling that follows the music: tight on dnb, generous on slow reggae.

The previous fixed 2 s window from M11c.3b cut off slow-music users (any ≤ 90 BPM with a 4-tap minimum exceeded the ceiling), silently firing Set the 1 instead of tap tempo. Reggae and slow hip-hop are in the target audience, so this had to go.

Rolling preview during the session (Traktor pattern)

From tap 3 onward, the deck-header BPM column shows 60 / median(tap_intervals_so_far), updated after each new tap. This is a rough live estimate, not the committed value — visually distinguished (italic, accent-orange, "TAPPING · N" badge) so the user knows it isn't final. On commit (constrained re-analysis completes) the deck header replaces the preview with the precise value AND the library row updates to match, both in the same publisher tick.

The library row does NOT show the preview. Library row updates only on commit. This solves the "I'm trying to fix my last wrong tap but the wrong BPM is still on the screen" mental-model breakdown from the prior UX round.

Recovery path: "tap again" fixes any tap mistake

If the user taps a wrong BPM (misheard the kick on a syncopated hi-hat, latched onto the off-beat, picked the wrong octave on a half-time tune), the recovery gesture is identical to the original gesture: tap again. Each tap session is fully independent.

  • New session's tap median is the only BPM hint. Previous user_tap BPM is not consulted, blended, or used as a fallback.
  • New session's first tap is the only anchor seed.
  • New session writes a new user_tap row; demotes the previous user_tap row for this track to is_active = 0 in the same transaction.
  • Demoted rows stay in the table for audit. v1 ships no "revert" UI; the data is ready for it.

The function signature enforces this: analyze_beat_grid_from_taps(samples, sr, ch, tap_times, profile). No "previous BPM" parameter. No "previous tap session" parameter. There is no way for prior tap state to leak into the new computation.

7 · How auto-analysis arrives at the grid

The estimator already shipped; this is the contract between its stages and the data model.

Six stages, two outputs (BPM/anchor on top row, bar_phase on bottom row), one combined BeatGrid. Integer-BPM snap fires only when the LSQ residuals don't get worse — protects 0.5-increment tempos (87.5, 128.5) from being clobbered.

8 · Drift — what we tell the DJ when uniform isn't enough

Vintage soul, live-band reggae, drummer-played breakbeats will not stay grid-locked over a long mix. Dub is honest about that: the grid stays uniform, the ⚠ indicator warns, and the mitigation lives on the pitch slider. Per PRD §8.3.3, warp markers are explicitly out of v1.

First downbeat (2 s) lands square on the kick. By the third downbeat (6 s) the kick is visibly ahead of the grid; by the fourth, the displacement is ~50 ms. The ⚠ "May drift" indicator surfaces when the LSQ drift slope crosses 3 ms/min. We do not invent fake beats; we tell the truth and let the DJ mix.

9 · How we compare against the industry

All four major DJ apps ship a version of the same uniform-4/4 contract. The differences are mostly UX ceiling — how much the GUI lets the user push beyond uniform. Dub takes the Serato bar-rotation model, runs the full estimator on every tap-tempo gesture (further than Traktor's hint reconcile), enforces lock-is-absolute end-to-end, and adds integer-BPM snapping by default for dance music.

App Tempo model Set-the-1 gesture Warp markers
Serato Single BPM per track Drag BPM box · BAR key rotates downbeat Yes (BAR markers post-2018)
rekordbox Single BPM + downbeat anchor Auto-grid + memory cue correction Yes (memory cue + per-region BPM)
Traktor Single BPM per track Manual nudge keys for grid offset Limited (Auto-Grid Lock)
Engine DJ Single BPM + first downbeat Beatgrid editor (drag) No (perform-time grid is read-only)
Dub (v1) Single BPM + arbitrary-position anchor + explicit bar_phase 1 tap = pure phase rotation · 3+ taps = constrained re-analysis (tap is search hint, not BPM) No (parked for v2)

Where we lean Serato

  • Bar-rotation: 1 tap = pure phase rotation, never moves beats.
  • Tap-tempo writes a user_tap row as the active grid; previous grids preserved for revert.
  • Grid lock = "this is the truth, don't touch it."

Where we go further than Traktor

  • Traktor's "Set Beatgrid" reconciles tap-derived BPM with the existing one inside a tolerance band. Dub instead runs the full estimator again with the tap median as a search constraint — same precision as auto-analyze, with the user's hint disambiguating octaves.
  • Strict no-warp: drift indicator instead of multi-anchor grids.

Where we go our own way

  • Integer-BPM snap by default — dance music dominates the target catalog.
  • Adaptive noise floor on transient snap so the 1 never lands on silence.
  • Lock is absolute. No "force re-analyze" bypass. A locked grid is a frozen contract; the user must explicitly unlock first. Silently working around a user-set lock is the same trust violation as silently re-pitching a track.
  • Library row and deck-header BPM are the same value, always. One source of truth, two observers. Most apps cheat on this (rekordbox shows the original BPM in the library while the deck shows the corrected one); Dub does not.
  • No mouse-driven grid editing in v1. Tap-to-grid + lock toggle covers the contract for the urban / scratch DJ.

Where we say "not v1"

  • Warp markers / multi-anchor flex grids — v2, gated by user demand.
  • Time-signature support beyond 4/4 — v2.
  • Per-section BPMs for tempo-changing songs — v2.

10 · Reliability gates

Anything that contradicts these is a bug. They restate PRD §1, §2.2.6, §8.3.3 in the local terms of this spec.

  1. Uniform 4/4 only in v1. No warp, no tempo curves, no swing modeling, no half-bars.
  2. The grid is the whole track. No code path may trim beats, advance the anchor past silence, or render fewer beats than [0, duration] implies.
  3. bar_phase is independent. Never inferred from beats[0], "first audible," or "first tap" outside the explicit tap-tempo rule.
  4. grid_locked = true is absolute. No code path — analyze, re-analyze, tap-tempo, set-the-1, anchor nudge, BPM nudge — may mutate bpm, beat_anchor_secs, or bar_phase while locked. The previous force re-analyze parameter is removed. See §11.
  5. Unlocking is an explicit gesture. Right-click Lock grid (to unlock) before any grid edit. No combined "unlock-and-do-X" affordances; lock toggle is its own audit event.
  6. Re-analyze on an unlocked track demotes the active user_tap row. Previous tap-derived grid stays in the table for audit.
  7. Each tap-tempo session demotes any prior user_tap row for the same track and is fully independent of all prior tap sessions. New session's tap median is the only BPM hint; prior user_tap BPM is never consulted, blended, or used as a fallback. Only one user_tap row per track is active at a time. The user's recovery path for any tap mistake is "tap again."
  8. Tap input is rejected on locked grids before crossing the FFI boundary. Swift gates on deckState.gridLocked; the Rust engine independently rejects (defense in depth). The deck-header BPM column visually signals "tap disabled."
  9. Tap tempo (3+ taps) requires deck transport playing. The first 1–2 taps are accepted in any transport state (Set the 1 path); a 3rd tap on a paused / cued / stopped deck is rejected with a brief "Press play to tap tempo" hint. Set the 1 is unaffected by transport state.
  10. Lock toggle is intentionally a mouse-only / prep-mode gesture. No keyboard shortcut. Right-click Lock grid on the library row or the loaded deck is the only entry point. Lock is "I'm done editing this grid" — a prep gesture, not a performance gesture.
  11. Tap session window is dynamic. max(1.5 s, 1.5 × median tap interval so far) after each tap. Replaces the fixed 2 s window that cut off slow-music users.
  12. Deck-header BPM column shows a rolling preview during an active tap session from tap 3 onward. Library row stays at the committed BPM until commit. Preview is visually distinguished (italic + accent + "TAPPING · N" badge) so the user never confuses it with a final value.
  13. Right-click menu uses one Analyze / Re-analyze entry whose label and enabled state follow §11. Parallel entries are removed.
  14. Library row BPM and deck-header BPM are always in sync for the loaded track. Single source of truth: the active track_beatgrids row. See §12.
  15. Analyze pre-renders the waveform sidecar at ~/Library/Caches/Dub/waveforms/{fingerprint}.wf. Deck load reads it synchronously and paints before the first frame. "Waveform building…" on a track with a sidecar is a regression. See §12.
  16. Drift > 5 % gets the ⚠ indicator. Pitch slider is the mitigation.
  17. The audio thread never touches grid edits. All grid mutation is off-RT; the audio thread reads an atomic snapshot.

11 · Lock semantics and the right-click menu

Lock is absolute, by design. We don't ship a "force" bypass. The user explicitly unlocks before any grid edit. The right-click menu reflects the lock state through one entry whose label switches on whether the track has been analyzed before.

Track stateMenu labelEnabled?Tooltip on hover
Never analyzed (no row in track_beatgrids) Analyze Yes —
Has analysis, unlocked Re-analyze Yes —
Has analysis, locked Re-analyze No "Unlock the grid first"

Why two clicks instead of a combined "unlock and re-analyze"

Combining the gestures hides the lock-state change. The lock toggle is its own observable event in the track_beatgrids audit trail; a DJ who unlocks intentionally and a DJ who re-analyzes intentionally are doing two different things, and the system records each. Two clicks is the price of keeping the audit trail honest.

12 · BPM parity and the waveform sidecar

The BPM in the library row and the BPM in the deck header are always the same number, sourced from the same active track_beatgrids row. The waveform sidecar is the parallel guarantee for the visual: analyze writes it, deck load reads it synchronously, the first frame paints with peaks already in place. "Loading…" on a track with a cache is a bug.

BPM parity — what we guarantee

  • Library has a BPM → deck header shows it instantly on load. No nil-and-wait.
  • Auto analyze completes → library row publishes → both bindings refresh same frame.
  • Tap-tempo commits → both update together.
  • Re-analyze commits → both update together.
  • Lock toggle → BPM unchanged → neither display changes.

Waveform sidecar — what we guarantee

  • Analyze and Re-analyze write ~/Library/Caches/Dub/waveforms/{fingerprint}.wf before returning.
  • DeckState.loadTrack reads the sidecar synchronously when present.
  • Deck waveform paints with sidecar data on the first frame; live PeakBuffer takes over after.
  • "Waveform building…" on a track that has a sidecar is a regression against this spec.
